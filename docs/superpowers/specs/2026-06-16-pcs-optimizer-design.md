# パワコン自動構成（下書き補助）設計

- 日付: 2026-06-16
- 対象: ソーラーレイアウト設計支援アプリ（`solar-layout`）／③パワコン構成画面
- ブランチ: `claude/solar-panel-layout-vrqcs4`

## 1. 目的・位置づけ

ゼロから手でパワコン構成（各台の型式・直列数・並列・分岐）を組むのは手間がかかる。
本機能は「機種・台数・1台あたり最大回路数」を指定して**最適化ボタン**を押すと、
図面の実パネル在庫に合わせた構成の**下書き**を自動生成する補助である。

**重要な前提:** これは最終採用案を出す機能ではない。
「ほぼ出来上がり、あとは手で微調整すれば完成」という段階まで自動で持っていくのが目的。
よって出力は必ず通常の編集可能なユニット（`plant.pcsUnits`）へ流し込み、
要調整箇所（残枚数・短い回路・電流超過の警報・必要台数）が一目で分かることを重視する。

## 2. 決定済みの方針

- ゴール優先順位: **電圧成立（絶対）→ 図面パネルの使い切り → 台数間のバランス**。
- 過積載率（DC/AC）の**上限は設けない**。
- **電流（アンペア）超過は容認**する（エラーにしない）。ただし**確認警報は必ず表示**する。
- 電圧上限を超える直列数は**そもそも生成しない**（絶対条件）。
- 1台で複数型式を許す（**MPPTごとに別型式**）。Huaweiはマルチmpptのため可能。
- 分岐ケーブル＝同一MPPTに並列接続。**分岐したストリングは同一型式・同一直列数**でなければならない
  → データ構造上は `PcsString.parallel ≥ 2`（同一 panelId・series を束ねる）で自然に表現できる。
- 台数は**固定**。入る分だけ割り振り、入り切らない分は**残枚数**として集計し
  「使い切るにはあと約N台」を提示する。
- 最適化の入力は **機種・台数・1台あたり最大回路数（手入力／既定=機種のMPPT数）** の3つ。
  最大回路数はマスタに保存しない（その都度入力。Huaweiなら手で 3 に直す運用）。
- 反映方式は **プレビュー → 適用**。プレビューで結果（台数・各台の型式/直列/並列・残・警告）を
  確認し、「適用」で `plant.pcsUnits` を置き換える。「破棄」で取り消し。

## 3. UI

パワコン構成画面のサマリカード内に「🎯 自動構成（下書き）」ブロックを追加する。

入力:
- 対象パワコン機種（マスタから選択。既定=先頭）
- 台数（整数 ≥ 1）
- 1台あたり最大回路数（整数 ≥ 1。既定 = 選択機種の `mpptCount`）

操作:
- 「最適化（下書きを作成）」ボタン → 下にプレビュー領域を表示。
- プレビューに「この内容を適用」「破棄」ボタン。
- 台数0・最大回路数0・機種未選択時はボタン無効＋注意文。

## 4. アルゴリズム（純関数 `optimizePcs`）

入力: 図面の型式別在庫、選択 PCS スペック、設計条件、台数 N、1台あたり最大回路数 C。

1. **在庫取得**: `summarizeLayout(layout, panels, "kaishu").byPanel` を型式別枚数として使う。
   各 model 名から該当 `PanelSpec` を引く。
2. **直列数の決定**: 型式ごとに `calcStringSizing(panel, pcs, conditions)` で
   有効直列範囲 `[seriesRange.min, seriesRange.max]` を得る（電圧上限＝絶対）。
   使い切りに有利になるよう、範囲内で「`枚数 mod series` が最小になる直列数」を選ぶ
   （同点なら直列数が大きい方＝回路数が少ない方）。
   `seriesRange.max < 1` または `min > max` の型式は「回路生成不可」として残へ。
3. **回路への分割**: 型式の枚数 T に対し、`full = floor(T / series)` 本の満数回路（各 series 枚）。
   端数 `R = T - full*series` が `R ≥ seriesRange.min` なら **短い回路**（R 枚）を1本追加、
   満たさなければ R 枚は**残**。
4. **台への配分（ラウンドロビン均等）**: 全型式の回路群を、各台のMPPT枠へ均等に詰める。
   制約:
   - 1 MPPT 行 = 単一型式・単一直列数。並列（分岐）数 ≤ `pcs.stringsPerMppt`。
   - 同一型式・同一直列数の回路は同じMPPT行に束ねて並列（分岐）として表現できる。
   - 1台のMPPT行数 ≤ `pcs.mpptCount`。
   - 1台の合計回路数（並列の総和）≤ C（最大回路数）。
   - 台ごとの合計回路数の差が最小になるよう順に割り当てる（バランス）。
5. **残の集計**: 制約で入り切らなかった回路の枚数＋手順3の端数残を合算し残枚数とする。
   `必要台数の目安 = ceil(総回路数 / C)` を算出して提示。

出力: `PcsUnitLine[]`（各台 `strings: PcsString[]`）、残枚数、必要台数目安、警告一覧。

## 5. データ反映

「適用」で `updatePlant(plant.id, { pcsUnits: 生成結果 })` により置き換える。
以後は既存の各台カードでそのまま手編集可能（＝微調整で仕上げ）。
プレビュー段階で全内容が見えているため、適用ボタン押下＝上書き確定とする。

## 6. 警告（容認するが知らせる）

- **電流超過**: ある回路で `parallel × panel.iscA > pcs.maxInputCurrentPerMpptA` のとき、
  エラー扱いにせず ⚠ 警報のみ表示（既存のストリングエラー判定とは別系統で「容認だが注意」と明示）。
- **短い回路**: 直列数が満数より少ない回路があれば注記。
- **残枚数 / あと約N台**: プレビューと適用後サマリに明示。
- 電圧超過になる直列は生成しないため、電圧系のエラーは原則発生しない。

## 7. エッジケース

- 図面にパネルが無い／全型式が直列下限を満たさない → 「作成できる回路がありません」。
- 台数0・最大回路数0 → ボタン無効＋注意文。
- 既存 `pcsUnits` がある状態で適用 → プレビューで確認済みのため一押しで上書き。
- `stringsPerMppt` が1（分岐不可機種）→ 1MPPT=1回路として配分（分岐ペアリングしない）。

## 8. 実装方針・テスト

- ロジックは純関数 `optimizePcs` として **新規 `src/calc/pcsOptimize.ts`** に分離する。
  UI（`PcsComposer.tsx`）はこの関数を呼んでプレビュー表示・適用するだけにし、薄く保つ。
- 単体テスト（`src/calc/pcsOptimize.test.ts`）で次を担保:
  - 単一型式・ぴったり割り切れるケース。
  - 複数型式の混在（MPPTごとに別型式）。
  - 端数が短い回路で吸収されるケース／吸収できず残になるケース。
  - 台数不足で残が発生し「あと約N台」が出るケース。
  - 分岐ペアリング（同一型式・同一直列を並列に束ねる）。
  - 電圧上限で直列数の上限が決まるケース。
  - 電流超過が警報になりエラーにならないこと。

## 9. スコープ外（YAGNI）

- 過積載率の上限制御・FIT区分連動の自動調整。
- 数理最適化ソルバ（線形計画等）の導入。
- 最大回路数のマスタ保存（その都度手入力で運用）。
- 複数機種を1回の最適化で混在させること（1回＝1機種）。
