# パワコン自動構成：配分の優先（影に強い／台数を詰める）設計

- 日付: 2026-06-16
- 対象: ソーラーレイアウト設計支援アプリ（`solar-layout`）／③パワコン構成「🎯 自動構成（下書き）」
- 前提: `docs/superpowers/specs/2026-06-16-pcs-optimizer-design.md`（自動構成の基本設計）の追補

## 1. 背景・目的

独立MPPT機（例 Huawei SUN2000-4.95KTL-NHL2：MPPT 2・1MPPTあたり分岐2並列）では、
同じパネル枚数でも「分岐2並列で1MPPTに束ねる」か「別々のMPPTに1並列ずつ散らす」かで
**部分影への強さ**と**入力電流**が変わる。

- 分岐2並列（同一MPPT）＝片方が影るとMPPTごと低下（ミスマッチ損）。電流も2倍（上限超過しやすい）。
- 別MPPTに1並列ずつ＝各MPPTが独立追従。影に強く、電流も素直。
- パワコンの能力（定格AC出力）は、接続DC合計が定格を上回っていれば**並列数に関係なく**発揮される。

現状の自動構成は「分岐優先」で詰めるため、同型が固まると分岐2並列（影に弱い・電流警報）になりやすい。
そこで**配分の優先をユーザーが選べる**ようにし、既定を「影に強い（分散）」にする。

## 2. 決定方針

- 自動構成に **「配分の優先」** セレクトを追加。選択肢は2つ：
  - **影に強い（MPPT分散）** … **既定**
  - **台数を詰める（分岐優先）**
- 既存の「機種・台数・最大回路数/台」はそのまま併用。
- 電圧上限は両モードとも絶対厳守。電流超過は容認＋警報（既存どおり）。
- 残枚数・「使い切るには約N台」も両モードで継続。
- 既定を「影に強い」にする理由：今の分岐2並列構成の弱点（影耐性・電流警報）を自動で解消できるため。

## 3. アルゴリズム（`distribute` に `strategy` を追加）

`strategy: "spread" | "dense"` を `distribute` と `optimizePcs` の引数に追加。
1回路ずつ配置するときの (a) 台の選び方 と (b) 1台内の置き方 を切り替える。

共通の制約（両モード）：
- 1台の合計回路数 ≤ `perUnitCap = min(maxCircuitsPerUnit, mpptCount * stringsPerMppt)`。
- 1 MPPT 行 = 単一 (panelId, series)・並列 ≤ `stringsPerMppt`。
- MPPT 行数 ≤ `mpptCount`。
- `multiMppt === false` の機種は1台＝単一 (panelId, series)（既存どおり）。

### spread（影に強い・既定）
- **台の選び方**：合計回路数が**最小**の台から（同点は番号小）＝バランスで薄く広げる。
- **1台内の置き方**：
  1. 空きMPPT行があれば**新しいMPPT行**に1並列で置く（別トラッカーへ分散）。
  2. 空きMPPT行が無い場合のみ、同一 (panelId, series) の既存行に**分岐（並列+1）**でフォールバック。
- 効果：同型の複数回路は極力別MPPT・別台へ散る。分岐は最後の手段なので電流警報も減る。
- 既知の制限：1台に空きMPPTが無い状態で別型式が来た場合、分岐で空けられないため**その台には同居できず別台**（または容量満杯なら残）になる。台数に余裕があれば残は出ない。使い切りは spread でも best-effort で優先するが、容量が逼迫した単一台ケースでは dense より置けないことがある。

### dense（台数を詰める・分岐優先）
- **台の選び方**：合計回路数が**最大**（=埋まっている）の、まだ上限未満の台から（同点は番号小）。空台は最後＝使う台数を最小化。
- **1台内の置き方**：
  1. 同一 (panelId, series) の既存行があれば**分岐（並列+1）**で束ねる。
  2. 無ければ**新しいMPPT行**。
- 効果：同型を分岐2並列でまとめ、最小台数に収める。指定「台数」は上限として働き、余った台は空のまま（結果に含めない）。

## 4. UI

「🎯 自動構成（下書き）」の入力行に **「配分の優先」セレクト** を追加（機種・台数・最大回路数の並び）：
- option: `影に強い（MPPT分散）`（value=spread・既定）／`台数を詰める（分岐優先）`（value=dense）
- 補足hintに一言：「分散＝影に強い・電流素直／詰める＝台数最少（分岐・コスト優先）」。

プレビューは既存のまま（各台のMPPT構成・枚数・残・警報・notes）。strategy により構成（並列1×2MPPT か 並列2×1MPPT か）が変わって表示される。

## 5. データ反映

`runOptimize` が `strategy` を `optimizePcs` に渡す。`applyOptimize` は従来どおり `plant.pcsUnits` を置換。
state に `optStrategy: "spread" | "dense"`（既定 `"spread"`）を追加。

## 6. テスト（`pcsOptimize.test.ts`）

`distribute` と `optimizePcs` に `strategy` 引数を追加するため、既存テストを更新し、モード別テストを追加する。

- **spread（既定）**
  - 同一型式の複数回路は、各台で**別MPPTに1並列ずつ**入る（空きMPPTがある間は分岐しない）。
  - 1台＋2型式（片方が2回路）では、同型が両MPPTを占めるため**もう片方は別台**（または容量満杯なら残）になる。
  - 空きMPPTが無い時のみ分岐にフォールバックすること。
- **dense**
  - 同一型式は**分岐（並列2）**に束ねられ、**最小台数**に詰まる（埋まった台から充填）。
  - 1台＋2型式は分岐で空きMPPTを作って**同居**できる。
- 既存の「分岐前提」テスト（例：単一台で2型式同居＝並列2を期待）は dense モードのテストへ振り替える、または spread の新挙動に合わせて期待値を更新する。
- 電圧上限・電流警報・残/必要台数の既存検証は維持。

## 7. スコープ外（YAGNI）

- 過積載率の上限制御（従来どおり無し）。
