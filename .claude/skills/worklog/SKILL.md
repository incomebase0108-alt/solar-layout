# Worklog — Git連動・自己要約ワークログ

4台のPCを行き来する作業で「直前の流れ」を引き継ぐための、Claude専用の軽量メモリー。
Slackの代わりに **Gitリポジトリ** を同期経路に使う（オフライン耐性・バージョン管理・接続品質に左右されない）。

## ファイル構成
- `.claude/worklog/CURRENT.md` — 直近のログ。**小さく保つ**（常時セッションに読み込まれる）。
- `.claude/worklog/archive/YYYY-MM.md` — 古いログを月次で圧縮要約したもの（必要時のみ読む）。
- `.claude/worklog/attachments/` — PDF・画像など。本文にはリンク＋1行要約のみ置く。

## このスキルでやること（3ステップ）

### ステップ1: 記録を追記する
今セッションの成果を **1ブロック** にまとめ、`CURRENT.md` の**先頭**（新しい順）に追記する。

```
## <YYYY-MM-DD HH:MM> — <machine> — <branch>
- やったこと: （箇条書きで簡潔に）
- 決めたこと: （設計判断・選択。無ければ省略可）
- 次の一手: （次のセッションが最初にやるべきこと）
- 添付: ./attachments/<file> （1行要約）  ← 添付がある場合のみ
```

- `<machine>` は環境変数 `CLAUDE_MACHINE` があればそれを、無ければ `hostname` の出力を使う。
- `<branch>` は現在の git ブランチ名。
- 1ブロックは簡潔に（5〜10行程度）。冗長な経緯は書かず、要点だけ。

### ステップ2: 肥大化していたら圧縮する（自己要約）
追記後、`CURRENT.md` のサイズを確認する。**閾値の目安: 約150行 または エントリ10個**。
超えていたら:
1. **最古のエントリ群**（古い方の数ブロック）を読む。
2. それらを**さらに短く圧縮要約**し、`archive/YYYY-MM.md` の末尾に追記する（1エントリ1〜2行）。
3. 圧縮した元エントリを `CURRENT.md` から**削除**する。
4. 結果として `CURRENT.md` は常に「直近の流れ」だけが残り、小さく保たれる。

> 圧縮は機械的な切り詰めではなく、**意味を保った要約**にすること。
> 「次の一手」など引き継ぎに不可欠な情報は残し、重複や過程の詳細を削る。

### ステップ3: 同期（コミット & プッシュ）
worklog 関連ファイルだけを個別にコミットして push する（コード変更とは分離）:
```
git add .claude/worklog
git commit -m "chore(worklog): <短い要約>"
git push -u origin <current-branch>
```
- push がネットワークエラーで失敗したら、指数バックオフ（2s, 4s, 8s, 16s）で最大4回リトライ。
- 繋がらない日は push をスキップしてよい（ローカルにコミットは残り、次回 SessionStart で同期される）。

## PDF・画像などの添付
- ファイルは `.claude/worklog/attachments/` に置き、`git add` してコミットする。
- 本文（CURRENT.md）には**リンクと1行要約だけ**を書く。本文に中身を貼り付けて肥大化させない。

## 注意
- CURRENT.md は「常時読む記憶」。詳細・過去経緯は archive と attachments（=必要時に読む記憶）へ逃がす。
- 同じ行を複数PCで奪い合わないよう、エントリは必ず**追記専用ブロック**として足す。
- 作業開始時は SessionStart フックが自動で `git pull` するので、通常は手動 pull 不要。
